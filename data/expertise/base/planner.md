## 기획자 기본 전문 지식

### 프로젝트 관리 원칙
- 요구사항은 SMART 기준(구체적, 측정 가능, 달성 가능, 관련성, 기한)으로 정의할 것
- 태스크 분배 시 각 담당자에게 완료 기준(Acceptance Criteria)을 명시할 것
- 의존성 있는 태스크는 반드시 depends_on으로 순서를 지정할 것
- 병렬 실행 가능한 태스크는 독립적으로 분리하여 효율을 높일 것

### 정보 설계(IA) 원칙
- 사이트맵은 3뎁스 이내를 권장, 4뎁스 이상은 사용자 이탈률 증가
- GNB(Global Navigation Bar)는 5~7개 항목이 최적 (밀러의 법칙)
- 콘텐츠 우선순위: 사용자 목적 기반으로 정보 위계를 설계할 것
- 레이블링은 사용자 언어로, 내부 용어 사용 금지

### 프로젝트 유형별 체크리스트
- **신규 구축**: 목적 정의 → 타겟 분석 → 벤치마킹 → IA 설계 → 콘텐츠 전략
- **리뉴얼**: 현황 분석 → 문제점 도출 → 개선 방향 → 유지/폐기 콘텐츠 분류
- **운영**: KPI 설정 → 데이터 기반 개선 → A/B 테스트 → 반복 최적화

### 인수 조건(Acceptance Criteria) — Gherkin 문법
- 모든 기능 요구사항은 반드시 Gherkin 문법(Given-When-Then)으로 인수 조건을 작성할 것
- **형식:**
  ```
  Feature: [기능명]
    Scenario: [시나리오 제목]
      Given [초기 상태 / 전제 조건]
      When  [사용자 행동 / 이벤트]
      Then  [기대 결과 / 검증 기준]
      And   [추가 결과 — 필요 시]
  ```
- 하나의 기능당 최소 2개 시나리오(정상 케이스 + 예외 케이스) 작성
- 기술 구현 방식이 아닌 **사용자 관점**에서 작성할 것
- `Given`은 테스트 시작 전 시스템 상태, `When`은 단일 사용자 행동, `Then`은 측정 가능한 결과로 작성
- 모호한 표현("적절히", "잘") 대신 구체적 기준(숫자, 상태값, 메시지)을 사용할 것

### 산출물 품질 기준
- 모든 기획 문서에 **목적**, **대상**, **범위**, **일정**을 명시할 것
- 태스크 설명에 "적절히", "잘" 같은 모호한 표현 사용 금지
- 디자이너/개발자에게 넘기는 요구사항은 해석의 여지가 없어야 함
- 인수 조건이 없는 기능 요구사항은 미완성으로 간주할 것
