## QA 기본 전문 지식

### 검수 기준 프레임워크
- **완전성**: 요구사항 대비 누락된 항목이 없는가
- **정확성**: 명시된 스펙(수치, 색상, 구조)과 일치하는가
- **일관성**: 문서/코드 전체에서 용어, 형식, 스타일이 통일되어 있는가
- **실현 가능성**: 기술적으로 구현 가능한 수준의 명세인가

### 기획 산출물 검수 포인트
- 사이트맵: 뎁스 일관성, 레이블 명확성, 빠진 페이지 없는지
- 요구사항: 모호한 표현("적절히", "잘") 사용 여부
- 태스크 분배: 각 태스크의 완료 기준이 측정 가능한지
- 의존성: 순환 의존 없는지, 누락된 선행 작업 없는지
- **Gherkin 인수 조건**: 아래 기준을 모두 충족해야 합격
  - Feature / Scenario / Given / When / Then 키워드가 올바른 계층으로 사용됐는지
  - 각 기능당 정상 케이스와 예외 케이스가 각 1개 이상 존재하는지
  - `Then` 절이 측정 가능한 결과(수치, 상태값, 메시지)로 기술됐는지
  - 구현 방법이 아닌 사용자 행동 관점으로 작성됐는지

### 디자인 산출물 검수 포인트
- 컬러: hex 코드 명시 여부, WCAG 대비율 충족 여부
- 타이포: 폰트명/크기/행간 명시 여부, 가독성 기준 충족
- 레이아웃: 그리드 일관성, 여백 체계 준수, 반응형 대응
- 상태 정의: default/hover/active/disabled 누락 없는지
- 접근성: 터치 타겟 크기, 포커스 표시, alt 텍스트 가이드

### 개발 산출물 검수 포인트
- HTML: 시맨틱 태그 사용, 접근성 속성(aria, role)
- CSS: !important 남용, 하드코딩된 매직 넘버, 반응형 대응
- JavaScript: 에러 핸들링, 메모리 누수 패턴, 이벤트 리스너 정리
- 성능: 이미지 최적화, 불필요한 리렌더링, 번들 크기

### 불합격 판정 원칙
- 요구사항 누락 1건 이상 → 불합격
- 스펙 불일치(컬러, 수치 등) → 불합격
- 모호한 표현으로 구현 불가능 → 불합격
- 사소한 개선 사항(nice-to-have) → 합격 후 부기사항으로 전달
